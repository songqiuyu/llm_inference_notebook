# GGML + GPT-2 架构全景解析

## 一、GGML 项目是什么

GGML 是一个**张量计算库**，专为在消费级硬件上高效运行大语言模型而设计。它的核心理念是：**用一套 C 代码定义计算图，通过编译期宏和多态后端，同时支持 CPU（AVX/NEON）、CUDA、Metal 等多种硬件**，无需修改一行推理代码。

整个项目的运转可以用一句话概括：

> 把神经网络表示成一张符号化计算图（DAG），通过兼容多后端的图级内存分配器管理显存/内存，再交给后端执行器按拓扑序逐节点计算。


## 三、main-batched.cpp 完整运转流程

### 阶段 0：模型加载（gpt2_model_load）

1. 从 GGML 格式文件中读入超参数（n_vocab=50257, n_ctx=1024, n_embd=768, n_head=12, n_layer=12）
2. 用 `no_alloc=true` 的轻量 ggml_context 创建所有权重张量的**空壳**（只有元数据，无 data）
3. 初始化后端：CUDA > Metal > CPU 自动选择
4. 在后端分配一块 buffer，把所有权重张量的 data 指针指向 buffer 中对应位置
5. 将文件中的权重数据直接读入（CPU/Metal）或间接拷贝到（CUDA）buffer
6. 为 KV cache 分配另一块 buffer（`n_layer × n_ctx × n_embd × 2` 个 float）

### 阶段 1：预分配计算内存（main 第 1036-1049 行）

```cpp
// 用最大可能 batch 构建"最坏情况"图，测算所需内存
batch.n_tokens = n_batch_max;
gf = gpt2_graph(model, batch, true);   // measure=true：按 n_ctx 估算 KV 大小
ggml_gallocr_reserve(allocr, gf);       // 模拟分配，计算 buffer 大小
```

只执行一次。之后每轮推理都复用同一块 buffer，不再 malloc。

### 阶段 2：首轮——处理 prompt（第 1056-1072 行）

```
prompt "Hello world" → tokenize → [15496, 995]
                                  ↓
            batch.token  = [15496, 995]
            batch.pos    = [0, 1]
            batch.seq_id = [0, 0]
            batch.logits = [false, true]   ← 只需最后一个 token 的 logits
                                  ↓
            gpt2_decode(model, allocr, batch, n_threads, logits)
                                  ↓
            logits 里有了 token[1] 位置对应的 50257 维 logit 向量
```

### 阶段 3：KV cache 共享（第 1074-1078 行）

```cpp
for (int i = 1; i < n_parallel; ++i)
    gpt2_kv_cache_seq_cp(cache, 0, i, 0, batch.n_tokens);
```

把 prompt 的 KV cache 标记为**同时属于所有并行序列**。这样 4 个序列共享同一个 prompt 的 KV，无需拷贝，零开销。

### 阶段 4：自回归生成循环（第 1108-1179 行）

```
while n_cur < n_len:
    for each 并行序列 i:
        logits_i = &logits[i_batch[i] * n_vocab]    // 取该序列上次的 logits
        new_id = sample(logits_i, top_k, top_p, temp) // 采样新 token
        batch.token[batch.n_tokens] = new_id         // 填入下一个 batch
        batch.pos   [batch.n_tokens] = n_cur
        batch.seq_id[batch.n_tokens] = i
        batch.logits[batch.n_tokens] = true
        i_batch[i] = batch.n_tokens++                // 记住下次去哪个位置取 logits

    gpt2_decode(model, allocr, batch, n_threads, logits)  // 一次前向，处理所有序列
    n_cur++
```

每轮循环只调用**一次** `gpt2_decode`，同时处理 n_parallel 个序列的 token，实现真正的批处理推理。

## 四、gpt2_decode 内部做了什么

这是整个推理的核心，每轮生成都调用一次：

```
1. 更新 KV cache 元数据
   └─ cache.cells[head + i] = {pos, seq_id}  for each token

2. 构建计算图
   └─ gf = gpt2_graph(model, batch, false)

3. 分配中间张量内存
   └─ ggml_gallocr_alloc_graph(allocr, gf)
      └─ 按仿真计划把 buffer 地址写入各 tensor->data
      └─ view 张量指向 view_src 的 data，不分配新内存

4. 设置输入数据
   └─ inp_tokens ← batch.token
   └─ position    ← batch.pos
   └─ KQ_mask     ← 构造因果掩码（-∞ 或 0）

5. 执行计算图
   └─ ggml_backend_graph_compute(model.backend, gf)
      └─ CPU: 多线程逐节点计算
      └─ CUDA: GPU kernel 逐节点执行

6. 提取输出
   └─ 从 gf 最后一个节点的 data 中读取 logits

7. 推进 KV cache head
   └─ cache.head += n_tokens
```

## 五、设计亮点

### 亮点 1：图算分离——先构图，再分配，最后执行

传统框架（如 PyTorch）是 eager 模式：写下 `c = a + b` 时立刻执行，立刻分配内存。GGML 采用**三段式分离**：

```
构建阶段（gpt2_graph）      分配阶段（gallocr）         执行阶段（compute）
─────────────────────      ──────────────────        ───────────────────
创建 ggml_tensor 节点    模拟执行→计算每个张量的    backend 遍历 nodes[]
只记录 op + src[] + 形状  生命周期→inplace 复用    逐节点调用 kernel
不分配 data 内存          → 最小化 buffer 大小      读 src->data → 写 dst->data
```

这意味着：
- **同一张图可以多次执行**，无需重新构图
- **内存分配只做一次**，后续轮次直接复用
- **图可以被序列化、优化、dump**，独立于执行

### 亮点 2：拓扑序生命周期分析 + Inplace 内存复用

`ggml_gallocr_alloc_graph_impl` 中，按拓扑序遍历每个节点：

```
对每个节点 node：
  1. 先分配所有 src（输入必须可用）
  2. 分配 node 本身的输出内存
     └─ 先尝试复用 parent：如果 parent 的引用计数为 1 且布局相同，
        直接让 node->data = parent->data，不分配新内存
  3. 每个 src 的 n_children -= 1，
     当 n_children == 0 且 n_views == 0 → 立即 free 该 src 的内存
```

**效果**：一个 12 层 GPT-2 的计算图有 200+ 个节点，如果没有复用，需要 ~200 块中间张量的内存。通过 inplace（add、mul、norm 等都支持）和 just-in-time free，实际只需要 **个位数 MB 的 buffer**。

打个比方：就像火车上的座位，每站有人上有人下，同一个座位被不同乘客轮流使用。200 个乘客只需要 10 个座位。

### 亮点 3：View 语义——零拷贝的张量重塑

GGML 中的 `view`、`reshape`、`permute`、`transpose` 都**不拷贝数据**，只创建一个新的张量对象，其 `data` 指针指向原张量的同一块内存，只是 `ne[]`（形状）和 `nb[]`（stride）不同：

```cpp
// 从 [2304, n_tokens] 的 cur 中切出 Q、K、V，零拷贝
Qcur = ggml_view_2d(ctx, cur, n_embd, n_tokens, cur->nb[1], 0*sizeof(float)*n_embd);   // 前 768 行
Kcur = ggml_view_2d(ctx, cur, n_embd, n_tokens, cur->nb[1], 1*sizeof(float)*n_embd);   // 中 768 行
Vcur = ggml_view_2d(ctx, cur, n_embd, n_tokens, cur->nb[1], 2*sizeof(float)*n_embd);   // 后 768 行

// Qcur、Kcur、Vcur 的 data 都指向 cur 的 data，只是 offset 不同
```

在 GPT-2 的 attention 计算中，Q/K/V split、reshape+permute、KQV merge 全都是 view 操作，一个字节都没拷贝。这让内存分配器可以把 view 标记为 `view_src` 的引用，只要源张量不被释放，view 就始终有效。

### 亮点 4：KV Cache 的 seq_id 集合——优雅的多序列共享

```cpp
struct gpt2_kv_cell {
    gpt2_pos pos = -1;
    std::set<gpt2_seq_id> seq_id;   // ← 关键设计：一个集合，不是一个 id

    bool has_seq_id(id) const { return seq_id.find(id) != seq_id.end(); }
};
```

每个 KV cache 条目可以**同时属于多个序列**。prompt 的 KV 通过 `gpt2_kv_cache_seq_cp` 被标记为 "seq 0 且 seq 1 且 seq 2 ..."，所有并行序列都能 attend 到这份共享的 prompt KV。

**好处**：
- 不需要为每个序列拷贝一份 KV（节省 `n_parallel × prompt_len × n_embd × n_layer` 的内存）
- 在 KQ_mask 中通过 `has_seq_id` 实现序列隔离，每个序列只能看到自己的生成 token
- 对比传统的 "每个序列独立 KV cache" 方案，这个设计在并行批处理场景下节省了大量显存

### 亮点 5：`no_alloc` Context ——张量元数据和数据分离

```cpp
// gpt2_graph 中的 ctx：只存元数据（ggml_tensor struct），不存数据
struct ggml_init_params params = {
    .mem_size   = buf_size,   // 刚好够放 4096 个 tensor 结构体
    .mem_buffer = buf.data(),
    .no_alloc   = true,       // ← 关键：不在 ctx 内分配 data
};

// gpt2_model_load 中的 ctx_w 同样 no_alloc：
// 权重张量的 data 由 ggml_backend_alloc_buffer 分配在后端 buffer 中
```

传统的 ggml 用法（`no_alloc=false`）是在 context 的 mem_buffer 里同时放元数据和数据。`no_alloc=true` 模式下：
- ggml_context 只管理张量的形状、类型、src 指针等**元数据**
- 真正的数据内存由**后端 buffer**（CPU RAM / GPU VRAM）管理

这意味着同样的计算图可以在 CPU 和 GPU 上无缝切换——换一个后端 buffer，把 data 指针指过去即可。

### 亮点 6：最坏情况预测量 + 一次分配永久复用

```cpp
// 用最大 batch 构建测量用图
batch.n_tokens = n_batch_max;
gf = gpt2_graph(model, batch, true);    // measure=true
ggml_gallocr_reserve(allocr, gf);        // 模拟执行，计算所需 buffer 大小
// ↑ 只做一次

// 之后的每轮推理：
ggml_gallocr_alloc_graph(allocr, gf);    // 复用同一块 buffer，直接设 tensor->data
```

整个推理过程中只在初始阶段做一次真正的 `malloc`（`ggml_backend_alloc_buffer`），之后几百轮自回归生成全部复用同一块 buffer。这在低延迟场景下极为关键——省掉了每轮的内存分配/释放开销。

### 亮点 7：后端多态——同一套代码跑所有硬件

```cpp
// main-batched.cpp 中后端选择的逻辑：
if (CUDA available)     backend = ggml_backend_cuda_init(0);
else if (Metal available) backend = ggml_backend_metal_init();
else                    backend = ggml_backend_cpu_init();

// 之后的所有操作与后端无关：
ggml_backend_tensor_set(inp_tokens, ...);     // CPU: memcpy, CUDA: cudaMemcpy
ggml_backend_graph_compute(backend, gf);       // CPU: 多线程, CUDA: GPU kernels
ggml_backend_tensor_get(inpL, ...);            // CPU: memcpy, CUDA: cudaMemcpy D2H
```

每个后端通过一个函数指针表（`ggml_backend_i`）暴露统一接口：
- `graph_compute` — 执行计算图
- `set_tensor` / `get_tensor` — 数据传输
- `alloc_buffer` / `free_buffer` — 内存管理

上层推理代码完全不用知道自己在 CPU 还是 GPU 上跑——后端切换对推理逻辑透明。

### 亮点 8：KQ_mask 的单头广播

KQ_mask 的形状是 `[n_kv, n_tokens, 1]`（第三个维度固定为 1），而不是 `[n_kv, n_tokens, n_head]`：

```cpp
KQ_mask = ggml_new_tensor_3d(ctx, F32, n_kv, n_tokens, 1);  // 只有 1 个头

// gpt2_decode 中：
for (int h = 0; h < 1; ++h) { ... }  // 只填充 1 份 mask

// gpt2_graph 中：
KQ = mul_mat(K, Q);                    // 形状 [n_kv, n_tokens, n_head]
KQ_masked = add(KQ_scaled, KQ_mask);   // [n_kv, n_tokens, n_head] + [n_kv, n_tokens, 1]
                                       // ↑ 自动广播到所有 head！
```

利用 ggml 的 broadcast 语义，1 份 mask 被广播到 12 个 head。如果 n_head=12、n_kv=2048、n_tokens=32，这省掉了 `2048 × 32 × 11 × 4 = 2.8MB` 的 mask 内存和填充开销。

### 亮点 9：compute flag 的懒标记

在 `ggml_visit_parents_graph` 中，`compute` 参数控制是否给节点打 `GGML_TENSOR_FLAG_COMPUTE` 标记。在 `ggml_build_forward_select` 中，只有被选中的那个输出节点及其依赖链上的节点会打 compute 标记，其他并行分支只做拓扑记录但不执行。

这允许一张图中同时包含多个输出分支，执行时可以选择性地只计算其中一个——对于多任务模型和 beam search 等场景非常有用。

### 亮点 10：整体设计哲学——用 C 的简单性驾驭 ML 的复杂性

整个 GGML 项目的核心代码量非常克制：
- `ggml.c` ~7000 行：图构建、反向传播、hash set
- `ggml-alloc.c` ~1100 行：图级内存分配器
- `ggml-backend.cpp` ~2300 行：后端抽象层
- CPU 算子实现分散在 `ops.cpp`（~8000 行）

没有任何第三方 ML 框架依赖。这套代码能跑 GPT-2、LLaMA、Whisper、Stable Diffusion 等多种模型。关键在于它把 ML 推理抽象成了三件事：

1. **张量运算** = 一个 DAG，节点是算子，边是数据依赖
2. **内存管理** = 按拓扑序分析生命周期，inplace 复用
3. **后端执行** = 一个 switch-case + 并行 kernel

这个极简抽象层让 GGML 可以在几乎所有平台上运行——从 x86 服务器到树莓派，从 NVIDIA GPU 到 Apple Silicon，代码改动极少。

## 六、一次完整推理的数据流

```
Prompt "Hello world"
     │
     ▼
Tokenizer ──→ [15496, 995]
     │
     ▼
┌───────────────── gpt2_decode ───────────────────┐
│                                                    │
│  inp_tokens ← [15496, 995]                         │
│  position   ← [0, 1]                               │
│                                                    │
│  WTE[15496] + WPE[0] ─┐                            │
│  WTE[995]   + WPE[1] ─┤                            │
│                        ▼                            │
│  ┌─────── Transformer Block 0 ──────────┐          │
│  │ LayerNorm → Q/K/V split              │          │
│  │   ├─ Q*K^T → scale → +KQ_mask        │          │
│  │   │          → softmax → ×V          │          │
│  │   ├─ K,V → ggml_cpy → KV_cache       │  ← 写入  │
│  │   └─ output projection → +residual   │          │
│  │ LayerNorm → FFN → GELU → +residual   │          │
│  └──────────────────────────────────────┘          │
│       ... 重复 11 次 ...                            │
│                                                    │
│  Final LayerNorm → lm_head ──→ logits [50257, 2]  │
│                                                    │
└────────────────────────────────────────────────────┘
     │
     ▼
  logits[:, -1]  → 采样 → next_token → 下一轮 batch
     │
     ▼
  KV cache head += 2, 继续循环...
```

## 七、关键创新总结

| 创新点 | 解决的问题 | 具体手段 |
|---|---|---|
| 符号执行图 | 跨硬件统一计算 | 先构图再执行，图与后端解耦 |
| 生命周期内存管理 | 中间张量占内存过大 | 拓扑序遍历 + 引用计数 + inplace |
| View 语义 | reshape/permute 拷贝开销 | 零拷贝，共享 data 指针 + 独立 stride |
| KV cache seq_id 集合 | 多序列 KV 重复存储 | 一个条目可被多个序列共享 |
| no_alloc context | 权重/计算内存分离管理 | 元数据在 ctx，数据在后端 buffer |
| 后端接口虚表 | CPU/CUDA/Metal 代码统一 | 函数指针多态，推理逻辑不感知硬件 |
| 单次预留 + 永久复用 | 每轮推理的 alloc 开销 | 最坏情况预分配，后续直接复用 |
